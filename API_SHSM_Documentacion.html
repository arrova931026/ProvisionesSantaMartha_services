1. Información general
ParámetroValor
URL basehttp://localhost:8080/api
Puerto8080
Context path/api
ProtocoloHTTP (REST/JSON)
AutenticaciónJWT Bearer Token (JJWT 0.12.6)
Formato de datosapplication/json (UTF-8)
FrameworkSpring Boot 3.4.0 · Java 21 · PostgreSQL 18
Roles del sistema:
• ADMIN – acceso total.
• AGENTE – gestiona contratos, cobros y pagos.
• CLIENTE – portal de autoservicio (lectura propia).
Manejo de errores: Todos los errores devuelven un objeto JSON con los campos timestamp, status, error y message. Errores de validación incluyen el campo errors (lista de campos inválidos).
2. Configuración de SoapUI
1
Crear proyecto REST
File → New REST Project → URI: http://localhost:8080/api
2
Crear propiedad de proyecto para el token
Clic derecho en el proyecto → Custom Properties → agregar propiedad:
Nombre: access_token   Valor: (vacío, se llenará al hacer login)
3
Agregar el token a cada request
En cada request REST, pestaña Headers → agregar:
Authorization = Bearer ${#Project#access_token}
4
Capturar el token automáticamente (TestCase)
Después del step de Login agrega un Groovy Script:
def response = context.expand('${Login#Response}') def json = new groovy.json.JsonSlurper().parseText(response) testRunner.testCase.testSuite.project.setPropertyValue( "access_token", json.accessToken )
5
Content-Type
Todos los requests con body deben incluir en Headers:
Content-Type: application/json
3. Módulo: Autenticación
POST /auth/login Obtener access token y refresh token

Público – no requiere token

Body (JSON):

{ "username": "admin", "password": "Admin123!" }

Respuesta 200:

{ "accessToken": "eyJhbGciOiJIUzI1NiJ9...", "refreshToken": "eyJhbGciOiJIUzI1NiJ9...", "username": "admin", "rol": "ADMIN" }
POST /auth/refresh Renovar el access token usando el refresh token

Público

{ "refreshToken": "eyJhbGciOiJIUzI1NiJ9..." }
POST /auth/logout Invalidar el refresh token

Requiere token

{ "refreshToken": "eyJhbGciOiJIUzI1NiJ9..." }

Respuesta: 204 No Content

11. Flujo de prueba recomendado en SoapUI
1
POST /auth/login – Obtén el token. Guárdalo en la propiedad del proyecto con el script Groovy.
2
GET /catalogos/metodos-pago – Anota el id del método "EFECTIVO" (o el que prefieras).
3
GET /planes – Anota el id del plan deseado (ej. Plan Básico = 1).
4
GET /catalogos/sucursales – Anota el id de la sucursal.
5
POST /personas – Crea el titular del contrato. Anota su id de la respuesta.
6
POST /contratos – Usa los IDs anteriores. El sistema generará los cobros mensuales automáticamente.
7
GET /cobros/contrato/{id} – Consulta los cobros generados. Anota el id del primer cobro.
8
POST /pagos – Registra el pago de la primera mensualidad usando el cobroId y metodoId.
9
GET /pagos/contrato/{id} – Verifica que el pago quedó registrado.
10
GET /notificaciones – Consulta si se generó alguna notificación.
12. Inserción del usuario administrador
⚠ La contraseña en la base de datos se almacena como hash BCrypt. Nunca se guarda en texto plano. Debes generar el hash de "Admin123!" antes de insertar.

Paso 1 — Generar el hash BCrypt de "Admin123!"

Ve a https://bcrypt.online, escribe Admin123!, rounds = 10, haz clic en "Hash Password" y copia el hash generado (empieza con $2a$10$).

Paso 2 — Ejecutar en pgAdmin (o psql) sobre la base de datos postgres, esquema funeraria

-- 1. Establece el esquema SET search_path TO funeraria, public; -- 2. Inserta la persona para el administrador INSERT INTO personas (nombre, ap_paterno, correo, pais) VALUES ('Administrador', 'Sistema', 'admin@shsm.com', 'México') RETURNING id; -- ► Anota el id devuelto (lo necesitas en el paso siguiente, ej: 1) -- 3. Obtén el id del rol ADMIN SELECT id FROM roles WHERE clave = 'ADMIN'; -- ► Anota ese id (normalmente 1) -- 4. Inserta el usuario admin -- Reemplaza: -- <PERSONA_ID> con el id obtenido en el paso 2 -- <ROL_ID> con el id obtenido en el paso 3 -- <BCRYPT_HASH> con el hash generado en el Paso 1 INSERT INTO usuarios (persona_id, rol_id, username, password_hash, requiere_cambio_pw, intentos_fallidos) VALUES (<PERSONA_ID>, <ROL_ID>, 'admin', '<BCRYPT_HASH>', false, 0) ON CONFLICT (username) DO NOTHING;

Ejemplo concreto (suponiendo persona_id = 1, rol_id = 1)

SET search_path TO funeraria, public; INSERT INTO personas (nombre, ap_paterno, correo, pais) VALUES ('Administrador', 'Sistema', 'admin@shsm.com', 'México'); INSERT INTO usuarios (persona_id, rol_id, username, password_hash, requiere_cambio_pw, intentos_fallidos) VALUES ( (SELECT id FROM personas WHERE correo = 'admin@shsm.com' LIMIT 1), (SELECT id FROM roles WHERE clave = 'ADMIN' LIMIT 1), 'admin', '$2a$10$REEMPLAZA_CON_TU_HASH_GENERADO_EN_BCRYPT_ONLINE', false, 0 ) ON CONFLICT (username) DO NOTHING;

Paso 3 — Verificar la inserción

SELECT u.id, u.username, r.clave AS rol, p.nombre, p.ap_paterno FROM usuarios u JOIN roles r ON r.id = u.rol_id JOIN personas p ON p.id = u.persona_id WHERE u.username = 'admin';

Paso 4 — Probar en SoapUI

POST http://localhost:8080/api/auth/login Content-Type: application/json { "username": "admin", "password": "Admin123!" }
Si recibes 401 Unauthorized, el hash no corresponde a "Admin123!". Vuelve a generarlo en bcrypt.online y actualiza con:

UPDATE funeraria.usuarios SET password_hash = '$2a$10$NUEVO_HASH' WHERE username = 'admin';
